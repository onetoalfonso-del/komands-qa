🗣 En palabras simples — ¿qué se construyó aquí?
Komands es una API que ServiceNow le manda a ON·NET para hacer operaciones sobre la red de fibra óptica: dar de baja un cliente, activar un acceso nuevo, hacer reset de un equipo ONT, etc. Esas órdenes van a través de un gateway (Axway APIM) y terminan en servidores Nokia o Huawei reales.

El problema era que no había una suite de QA confiable: la colección de Postman que existía mandaba los datos en un formato incorrecto, los tests de seguridad usaban nombres de permisos que ya no existían, y el pipeline de CI no corría bien. En la práctica, las pruebas "pasaban" pero no estaban probando el sistema real.

Lo que se hizo en este sprint fue construir toda esa infraestructura de QA desde cero: un servidor local que simula Komands, 351 tests automáticos, una colección Postman completa con reportes detallados, y un pipeline en GitHub que corre todo automáticamente cada vez que se sube código.

La especificación oficial del API Komands v2.2.3 (97 KB). Define exactamente qué endpoints existen, qué campos envía ServiceNow y qué responde Komands. Es la fuente de verdad — todo lo demás se escribió mirando este archivo.

mock_server.py
Un servidor web local (FastAPI) que imita el comportamiento de Komands. Responde en localhost:8000 con el mismo formato que el servidor real. MODIFICADO
tests/conftest.py
El archivo de configuración base que todos los tests importan. Contiene los "moldes" de datos (fixtures) que se usan en cada prueba — el formato de un request de baja, de activación, el token JWT de prueba, etc. Estaba usando formato incorrecto (campos planos en vez de family-format) y scopes viejos — se corrigió todo para alinearlo con el spec.
CREADO
tests/mocks/payloads.py
Librería de "datos de prueba malos" — payloads diseñados específicamente para provocar errores. Por ejemplo, si mandas ONT ID 8888 el mock responde "ONT no encontrado", si mandas serial FAIL00000000 simula un rollback. Sin esto, los tests solo verifican el caso exitoso y nunca saben si el sistema maneja bien los errores.
MODIFICADO
tests/security/test_feature_flags.py
Tests que verifican que la seguridad funciona bien: que alguien sin token no pueda hacer nada, que un token de solo-lectura no pueda crear órdenes, que un VNO no autorizado sea rechazado. Estaban usando el nombre de permiso komands:read que ya no existe — con ese nombre los tests pasaban por razones equivocadas. Se corrigió a komands:query.
REESCRITO
collection Kommand/SN a Kommand.postman_collection.json
La colección de Postman que el equipo usa para probar el API manualmente o con newman (CLI). Se reescribió completa: de formato incorrecto con requests que habrían fallado en producción a 10 carpetas, 30+ requests, family-format correcto, 7+ verificaciones por request y logs detallados en el reporte HTML. 🎫 ServiceNow

Sistema de origen de órdenes — integración TO-BE

¿Qué es?

ServiceNow es la plataforma que usan los operadores para gestionar órdenes de trabajo. Cuando un técnico hace una baja, una activación, o un cambio de equipo, lo registra aquí. ServiceNow traduce esa acción en un request HTTP que va hacia Komands.

¿Qué se hizo aquí en el sprint?

No se tocó ServiceNow directamente — es un sistema externo. Lo que se hizo fue documentar exactamente qué headers envía y construir la colección Postman para simular esos requests tal como los mandaría ServiceNow.

Headers que envía por request

X-Source-System: SN_PROVISION X-Correlation-ID: {uuid} X-User-ID: sn-operator-01 X-User-Organization: {VNO} Authorization: Bearer {JWT}

Los 11 endpoints que llama

POST/unsubscriptionDar de baja un servicio
POST/service-activationActivar acceso FTTH/SSAA nuevo
POST/service-modificationCambiar velocidad / bloquear / agregar VLAN
POST/device-modificationCambiar equipo ONT (swap)
POST/fiber-changeMover cliente a otro puerto/OLT
POST/reset-ontReiniciar equipo ONT del cliente
POST/pon-transferTraspaso entre puertos PON
POST/cancel-orderCancelar una orden en curso
GET/transaction/{id}/statusConsultar estado de una transacción
GET/access/{access_id}Consultar estado de un acceso
GET/port-occupancyVer ocupación del puerto PON

🔐 Axway APIM

API Gateway — portero de seguridad entre ServiceNow y Komands

¿Qué es?

Axway es el gateway que se para en el medio entre ServiceNow y Komands. Hace dos cosas: verifica que quien manda el request tiene permiso (OAuth2 con JWT) y controla quién puede hacer qué según su VNO y scope. Sin pasar por Axway, nadie puede llamar a Komands.

¿Qué se hizo aquí en el sprint?

No tenemos acceso a Axway todavía. Lo que se hizo fue recrear su comportamiento en el mock: el mock valida tokens JWT con los mismos scopes que Axway real, rechaza VNOs no autorizados, y retorna 401/403 de la misma forma.

Scopes del spec v2.2.3

komands:provision → POST (crear/modificar) komands:query → GET (solo lectura)

⚠ Antes del sprint se usaban nombres viejos (komands:write / komands:read) — se corrigieron en toda la suite.

⚡ Komands API v2.2.3

El cerebro de la operación — recibe órdenes y las ejecuta en las OLTs

¿Qué es?

Komands es el servidor central que recibe las órdenes de ServiceNow (vía Axway) y las traduce en comandos para los equipos Nokia y Huawei. Es asíncrono: cuando recibe una orden devuelve 202 Accepted inmediatamente y procesa la operación en background. El resultado final se puede consultar después con el endpoint de estado.

¿Qué se hizo aquí en el sprint?

Se versionó el spec oficial (openapi.json v2.2.3) en el repositorio — la primera vez que existe una fuente de verdad única accesible a todo el equipo. Se usó para corregir toda la suite de QA que estaba desalineada.

Formato de respuesta (siempre igual)

{ "result": { "u_uuid": "3fa85f64-...", // ID único de la transacción "u_return_code": "0", // 0 = éxito "u_return_code_desc":"Solicitud aceptada", "u_timestamp": "2026-06-16T...", "u_time": "0.001s" } }

Códigos de error KMD documentados

📡 Nokia OLTs — ISAM 7360

Equipos de red que conectan la fibra al cliente

¿Qué es?

Las OLTs (Optical Line Terminal) son los servidores de red instalados en las centrales telefónicas. Cada OLT conecta hasta 128 clientes a través de fibra óptica. Nokia ISAM 7360 es el modelo más común en la red de ON·NET.

¿Qué se hizo aquí en el sprint?

Se mapeó la diferencia clave entre Nokia y Huawei y se incorporó esa lógica al mock: Nokia no soporta REMOVE_SERVICE. Si un request intenta hacer esa operación en una OLT Nokia, el mock devuelve KMD-4001 igual que lo haría el servidor real.

Limitación importante

REMOVE_SERVICE → KMD-4001 (no soportado en Nokia)

Para eliminar un servicio en Nokia hay que hacer una baja completa (/unsubscription), no una modificación.

OLT-SAN-001OLT-VAL-001 OLT-VAL-002OLT-SCL-010

📻 Huawei OLTs — MA5800/MA5600T

Segunda familia de equipos de red — comportamiento distinto a Nokia

¿Qué es?

Otra familia de OLTs presentes en la red. Los Huawei MA5800 sí soportan eliminar un servicio específico sin dar de baja todo el acceso. Esta diferencia es importante porque el mismo request puede comportarse distinto según a qué OLT va.

¿Qué se hizo aquí en el sprint?

El mock diferencia por olt_name: si el nombre de OLT corresponde a un equipo Huawei (OLT-SAN-002, OLT-VAL-003), permite REMOVE_SERVICE. Si es Nokia, lo rechaza. Esta es la misma lógica que tiene el servidor real.

Diferencia clave vs Nokia

REMOVE_SERVICE soportado ✓ Serial format: 485754CXXXXX

📬 Postman Collection

collection Kommand/SN a Kommand.postman_collection.json — REESCRITA en este sprint

¿Qué es?

Una colección de Postman es un conjunto de requests HTTP organizados en carpetas. Con ella podés ejecutar todas las pruebas del API manualmente (desde Postman) o automáticamente (con newman desde la terminal). Es el entregable de QA principal que se puede mostrar al equipo como evidencia de que el sistema funciona.

¿Qué se hizo?

Se reescribió completamente. La versión anterior enviaba datos en formato incorrecto y habría fallado contra el servidor real. La nueva tiene 30+ requests en 10 carpetas, family-format correcto, 7 verificaciones individuales por request, y logs detallados visibles en el reporte HTML.

10 carpetas

01 · Baja de servicio 02 · Activación FTTH 03 · Modificación de servicio 04 · Cambio de equipo (swap) 05 · Cambio de fibra 06 · Reset ONT 07 · Traspaso PON ← nuevo 08 · Consultas síncronas 09 · Errores de autenticación ← nuevo 10 · Flujo E2E completo (6 pasos) ← nuevo

7 verificaciones automáticas por request

✅ HTTP 202 Accepted ✅ result.u_uuid es UUID válido (regex) ✅ result.u_return_code = "0" ✅ result.u_return_code_desc presente y no vacío ✅ result.u_timestamp presente ✅ result.u_time presente ✅ SLA < 5000ms

GitHub Actions es el sistema de integración continua (CI). Cada vez que alguien sube código o abre un Pull Request, GitHub ejecuta automáticamente todos los tests y el análisis de calidad. Si algo falla, el PR no puede mergearse. Así el equipo sabe en minutos si un cambio rompió algo.
