API REST para provision de servicios FTTH y SSAA sobre OLTs Nokia y Huawei.

Expuesta a ServiceNow SOM a traves de Axway APIM. Este mockup simula el comportamiento funcional documentado en el Documento de Diseno Tecnico v2.2.3.

Notas del mockup

  • Las operaciones async simulan procesamiento pero no ejecutan comandos CLI reales.
  • El callback HTTP POST al u_callback_url no se envia en esta version mockup.
  • El u_time en las respuestas refleja el tiempo de procesamiento interno (< 1ms en mockup).
  • La validacion Pydantic rechaza campos con tipos o valores enum incorrectos (HTTP 422).
  • No hay rate limiting implementado; en produccion lo gestiona Axway (20k tx/h global).

Autenticacion

Endpoint emisor de tokens JWT (OAuth 2.0 client_credentials). Komands emite el token; Axway actua como pass-through (TLS terminator + proxy). Las credenciales son emitidas por el equipo Komands para cada VNO y se gestionan en HashiCorp Vault. Ref: Seccion 5.2 del documento de diseno tecnico.

Emite un token JWT mediante flujo OAuth2 client_credentials (RFC 6749, seccion 4.4).

Opciones de envio de credenciales:

  • Form fields: client_id y client_secret como campos del formulario (Postman).
  • Basic Auth: Header Authorization: Basic Base64(client_id:client_secret) (Swagger UI).

Uso del token:

Authorization: Bearer <access_token>

Expiracion: 120 minutos (configurable). Claims: sub (client_id), scope (client), exp.

Ref: Seccion 5.2 del documento de diseno tecnico.

Health

Health checks de Kubernetes. Sin autenticacion. No se publican en Axway APIM.

  • /health — Liveness: ¿esta vivo el proceso?
  • /health/ready — Readiness: ¿puede recibir trafico? (PostgreSQL + Redis disponibles)

Ref: Seccion 6.3.

Baja de acceso

Baja logica de acceso en OLT. Body con familias u_routing + u_identification.

  • POST /unsubscription — Asincrono: respuesta 202 + callback a SN.
  • GET /unsubscription/{u_uuid} — Sincronico: consulta estado por UUID.

Ref: Secciones 2.1 y 2.2.

Consulta de acceso

Consulta sincrona del estado completo de un acceso en la OLT.

Response incluye bloque u_access con 23+ propiedades: identificacion, ubicacion, hardware, estado admin/oper, servicios habilitados, perfiles, VLANs, GEM ports, metricas opticas y fechas.

Formato {u_id}: OLT_SLOT_PON_ONT (ej: OLT-Santiago-01_1_3_10). Parser usa rsplit('_', 3) para soportar OLTs con guiones bajos en el nombre.

Ref: Seccion 2.3.

Modificacion de servicio

Habilita, deshabilita o modifica selectivamente IPTV, VOZ y GESTION. Body completo con las 8 familias jerarquicas.

u_action: A=Alta, B=Baja, M=Modificar. Flags u_iptv/u_voz/u_gestion: T=participa, F=excluido.

  • POST /service-modification — Asincrono: respuesta 202 + callback a SN.
  • GET /service-modification/{u_uuid} — Sincronico: consulta estado por UUID.

Ref: Seccion 2.5.

Cambio de fibra

Migracion atomica de un acceso desde PON antigua a PON nueva. Body con 8 familias + bloque adicional u_routing_new.

Komands ejecuta internamente: (1) activacion de prueba con serial ficticio en PON nueva, (2) baja en PON antigua, (3) Device Modification con serial real en PON nueva.

Codigos exclusivos: 110 (fallo paso 1, retryable), 115 (fallo paso 2 con rollback, retryable), 120 (estado inconsistente, NO retryable, escalar a MOS-iT).

  • POST /fiber-change — Asincrono: respuesta 202 + callback a SN.
  • GET /fiber-change/{u_uuid} — Sincronico: consulta estado por UUID.

Ref: Seccion 2.6.

Activacion de servicio

Activa servicios sobre una ONT ya provisionada. Body con 7 familias (sin u_hardware).

Comportamiento por producto:

  • FTTH: Activa servicios indicados por flags u_iptv/u_voz/u_gestion (al menos uno T).

  • SSAA: Activa automaticamente INTERNET y TRUNKIP, ignora flags.

  • POST /service-activation — Asincrono: respuesta 202 + callback a SN.

  • GET /service-activation/{u_uuid} — Sincronico: consulta estado por UUID.

Ref: Seccion 2.7.

Cambio de Pelo

Migra un acceso a una nueva PON manteniendo todos los servicios activos. Incluye el payload completo de servicios (VLANs, GEM ports, perfiles) en una sola peticion.

El vendor (Huawei/Nokia) lo resuelve el orquestador desde la OLT. La validacion de campos minimos (serialnumber, speed_plan, inet_svlan + services_port Internet, y por cada flag en T su SVLAN + entrada en services_port) se realiza a nivel de API.

  • POST /api/Komands/v1/pon-transfer — Asincrono: respuesta 202 + callback a SN.
  • GET /api/Komands/v1/pon-transfer/{u_uuid} — Sincronico: consulta estado por UUID.

Ref: Seccion 2.8.
