[066] Versión revisada (Mayo 2026) — Alineamiento con KOMANDs API v1 v0.3.0 (Anexo H del LLD Sunset BP v2.1). Cambios principales aplicados:
[067] Endpoints homologados al patrón canónico /api/Komands/v1/<recurso> (se eliminaron variantes /apiKomands/v1/, /api/v1/modification y el endpoint inexistente /api/v1/transaction/{uuid}).
[068] Nomenclatura de campos JSON alineada con el contrato KOMANDs (prefijo u_): u_olt, u_slot, u_pon, u_ontid, u_serialnumber, u_slid, u_uuid, etc. Campo de VNO renombrado a id_vno (enum: DTV, CVTR, ENTEL, TCH).
[069] Política de reintentos formalizada con 5 system properties (kill switch, callback timeout, max attempts, backoff intervals, retryable codes). Los intervalos 0/5/15/45/120s pasan a ser configurables.
[070] Lógica de retry migrada de categorías simbólicas (RATE_LIMIT/SSH_ERROR/CLI_TIMEOUT) a códigos funcionales numéricos de KOMANDs (u_return_code 40, 50) y códigos HTTP (429, 502, 503).
[071] Header Authorization: Bearer <jwt> declarado explícitamente como obligatorio en Outbound REST Messages, inyectado dinámicamente vía global.AxWayUtils.getToken().
[072] Renombrado de identificador de transacción: txn_id → u_uuid (alineado con el campo emitido por KOMANDs en el ACK 202 y en el header X-Transaction-ID).
[073] Corrección de typos ("maxma" → "máxima", "Kommands" → "Komands").
[074] Referencia Normativa al Contrato KOMANDs API v1
[075] Este documento describe la implementación en ServiceNow para consumir las APIs publicadas por la plataforma KOMANDs. La especificación contractual de las APIs (paths, métodos, schemas, headers obligatorios, códigos de error, políticas de rate limiting y SLOs) es normativa en el documento "KOMANDs API v1 — Diseño Técnico v0.3.0" (en adelante "KOMANDs Spec").
[076] Regla de precedencia: ante cualquier conflicto entre este documento y KOMANDs Spec, prevalece KOMANDs Spec. Toda divergencia detectada debe registrarse como hallazgo y resolverse antes de la implementación, con aprobación cruzada de los equipos MOS-iT (KOMANDs), ServiceNow y ON·NET (Axway).
[077] Las subsecciones siguientes embeben las definiciones contractuales clave de KOMANDs Spec con la referencia a la sección original. Esta duplicación es intencional para que un desarrollador de ServiceNow tenga toda la información operativa en un único documento.

[421] Fin de Referencia Normativa al Contrato KOMANDs API v1. Las siguientes secciones del documento describen la implementación específica en ServiceNow.
[422] Objetivo
[423] Eliminar las dependencias de Blue Planet en los flujos actuales y migrar la integración a:
[424] CMDB (ServiceNow) como fuente de datos, cuando antes se consultaba Blue Planet para obtener información.
[425] Komands como sistema destino para ejecutar bajas y modificaciones, cuando antes se enviaban solicitudes a Blue Planet.
[426] Enfoque de implementación
[427] Implementación de flujos paralelos v2 para el desacoplamiento total de los módulos BPI / BPO (eliminando dependencia con Blue Planet).
[428] Duplicar las tablas de core use case, por ejemplo, u_assignment con u_assignment_v2 o crear las tablas necesarias para soportar la nueva integración como u_komands_transaction .
[429] Eliminar el uso de la tabla u_control_integration .
[430] Al existir versiones sincronas y asincronas y versiones tanto para FFTH como para SSAA por separados, se crearán nuevas versiones para el uso de Komands de igual forma.
[431] Agregar una validación a los endpoints de las API para los feature flags. Dependiendo del resultado se crea el registro en la tabla original (con BluePlanet) o la nueva tabla relacionada a los flujos Komands (v2).
[432] En cada flujo duplicado, clasificar cada dependencia de Blue Planet en uno de estos dos tipos y reemplazarla:
[433] Tipo A — Peticiones operacionales (baja/modificación/deconfiguración): 
[434] Reemplazar las llamadas a Blue Planet por llamadas a Komands.
[435] Tipo B — Obtención de datos (lookups/queries): 
[436] Reemplazar las llamadas a Blue Planet por queries a CMDB (tablas y relaciones existentes en ServiceNow).
[437] Estandarizar el patrón asíncrono:
[438] el flujo realiza un Wait for Callback hasta que exista una respuesta final en u_komands_transaction,
[439] luego delega la espera, manejo de timeouts y errores a un subflujo reutilizable: Integration Control Komands
[440] Convención: acciones dedicadas para Komands
[441] Cada request a Komands debe implementarse como una Action propia (reutilizable) para evitar lógica duplicada en los flujos.
[442] Formato de nombre (estándar) 
[443] <API o Proceso> - Komands
[444] Ejemplos
[445] Access Deregistration - Komands
[446] Device Modification - Komands
[447] Qué incluye la Action
[448] Mapeo de datos y construcción del esquema JSON conforme a la especificación de la API.
[449] Llamada HTTP/REST a Komands.
[450] Manejo de respuesta inmediata (ACK) y extracción de correlation_id.
[451] Salida estandarizada para el flujo (komands_transaction record).
[452] Qué NO incluye la Action
[453] El Wait for Callback notification por respuesta final en u_komands_transaction (esto queda en el subflujo Integration Control Komands).
[454] El parseo/interpretación de la respuesta final y las actualizaciones de records de negocio (esto queda en el subflujo Integration Control Komands).
[455] Callback inbound (Komands → ServiceNow)
[456] Para registrar la respuesta final de operaciones asíncronas, se creará un Scripted REST API inbound (webhook) con nombre Komands Callback API
[457] /api/insp9/komands_callback
[458] Comportamiento esperado
[459] Komands invoca el endpoint de callback cuando la transacción finaliza.
[460] ServiceNow actualiza u_komands_transaction usando solo u_txn_id como llave de correlación.
[461] El id de transacción se obtiene desde el ACK de la primera llamada a Komands y queda guardado en el flujo/registro.
[462] Política de reintentos (Wait / consulta de estado de transacción)
[463] La política de reintentos de Komands se basa en backoff exponencial alineado con KOMANDs API v1 sec. 3.4.2. Los valores son parámetros configurables vía System Properties (no hardcoded) — ver subsección "Crear system properties". Los valores por defecto son los siguientes:
[464] Reintento 1: inmediato
[465] Reintento 2: 5s
[466] Reintento 3: 15s
[467] Reintento 4: 45s

[979] Propuesta inicial Feature Flags
[980] El mecanismo de enrutamiento hacia Komands se basa en una tabla de configuración u_komands_activation, que define por combinación de VNO, producto, tecnología y operación si una transacción debe ser procesada por Komands o por BluePlanet.
[981] Evaluación de activación por transacción
[982] Cada vez que una API recibe una solicitud, consulta u_komands_activation mediante un Script Include.
[983] Si existe un registro activo para la combinación consultada (u_is_active = true), la transacción se dirige a Komands (por ejemplo, insertando en u_assignment_v2).
[984] Si no existe un registro activo, la transacción se procesa por la ruta tradicional en BluePlanet (por ejemplo, insertando en u_assignment).
[985] Kill switch global
[986] Existe un kill switch global u.komands.enabled que permite desactivar completamente el uso de Komands en cualquier momento, forzando que todo el tráfico vuelva a BluePlanet sin necesidad de despliegues ni cambios de código.
[987] Consideraciones de diseño
[988] Este mecanismo está pensado como una solución temporal para habilitar una migración progresiva, por lo que no se prioriza un diseño altamente escalable. En un escenario permanente, sería recomendable modelar la activación de cada dimensión (VNO, producto, tecnología, operación) en estructuras separadas o más normalizadas, permitiendo evaluaciones más eficientes y flexibles. El enfoque actual favorece la simplicidad operativa y la capacidad de rollback inmediato.
