[000] Sunset BluePlanet - Desarrollo ServiceNow
[001] Objetivo
[002] Eliminar las dependencias de Blue Planet en los flujos actuales y migrar la integración a:
[003] CMDB (ServiceNow) como fuente de datos, cuando antes se consultaba Blue Planet para obtener información.
[004] Komands como sistema destino para ejecutar bajas y modificaciones, cuando antes se enviaban solicitudes a Blue Planet.
[005] Enfoque de implementación
[006] Implementación de flujos paralelos v2 para el desacoplamiento total de los módulos BPI / BPO (eliminando dependencia con Blue Planet).
[007] Duplicar las tablas de core use case, por ejemplo, u_assignment con u_assignment_v2 o crear las tablas necesarias para soportar la nueva integración como u_komands_transaction .
[008] Eliminar el uso de la tabla u_control_integration .
[009] Al existir versiones sincronas y asincronas y versiones tanto para FFTH como para SSAA por separados, se crearán nuevas versiones para el uso de Komands de igual forma.
[010] Agregar una validación a los endpoints de las API para los feature flags. Dependiendo del resultado se crea el registro en la tabla original (con BluePlanet) o la nueva tabla relacionada a los flujos Komands (v2).
[011] En cada flujo duplicado, clasificar cada dependencia de Blue Planet en uno de estos dos tipos y reemplazarla:
[012] Tipo A — Peticiones operacionales (baja/modificación/deconfiguración): 
[013] Reemplazar las llamadas a Blue Planet por llamadas a Komands.
[014] Tipo B — Obtención de datos (lookups/queries): 
[015] Reemplazar las llamadas a Blue Planet por queries a CMDB (tablas y relaciones existentes en ServiceNow).
[016] Estandarizar el patrón asíncrono:
[017] el flujo realiza un Wait for Callback hasta que exista una respuesta final en u_komands_transaction,
[018] luego delega la espera, manejo de timeouts y errores a un subflujo reutilizable: Integration Control Komands
[019] Convención: acciones dedicadas para Komands
[020] Cada request a Komands debe implementarse como una Action propia (reutilizable) para evitar lógica duplicada en los flujos.
[021] Formato de nombre (estándar) 
[022] <API o Proceso> - Komands
[023] Ejemplos
[024] Access Deregistration - Komands
[025] Device Modification - Komands
[026] Qué incluye la Action
[027] Mapeo de datos y construcción del esquema JSON conforme a la especificación de la API.
[028] Llamada HTTP/REST a Komands.
[029] Manejo de respuesta inmediata (ACK) y extracción de correlation_id.
[030] Salida estandarizada para el flujo (ej. correlation_id, status_code, response_body).
[031] Qué NO incluye la Action
[032] El Wait for Callback notification por respuesta final en u_komands_transaction (esto queda en el subflujo Integration Control Komands).
[033] El parseo/interpretación de la respuesta final y las actualizaciones de records de negocio (esto queda en el subflujo Integration Control Komands).
[034] Callback inbound (Komands → ServiceNow)
[035] Para registrar la respuesta final de operaciones asíncronas, se creará un Scripted REST API inbound (webhook) con nombre definitivo:
[036] api/v1/komands/callback
[037] Comportamiento esperado
[038] Komands invoca el endpoint de callback cuando la transacción finaliza.
[039] ServiceNow actualiza u_komands_transaction usando solo txn_id como llave de correlación.
[040] El txn_id se obtiene desde el ACK de la primera llamada a Komands y queda guardado en el flujo/registro.
[041] Política de reintentos (Wait / consulta de estado de transacción)
[042] La política de reintentos de Komands es la siguiente:
[043] Reintento 1: inmediato
[044] Reintento 2: 5s
[045] Reintento 3: 15s
[046] Reintento 4: 45s
[047] Reintento 5 (último): 120s
[048] Nota
[049] ServiceNow puede consultar el estado vía GET /api/v1/transaction/{uuid} como mecanismo de reconciliación. Para ello se deberá crear una Action adicional por operación (Get (API) Status - Komands) y utilizarla pasado los 120s.

[076] Subflujo reutilizable: Integration Control Komands (nuevo comportamiento)
[077] Subflujo reutilizable para estandarizar el patrón asíncrono con Komands.
[078] Objetivo
[079] Recibir una transacción (u_komands_transaction) y ejecutar el Wait for Callback dentro del subflujo.
[080] Manejar timeouts y errores de callback dentro del subflujo.
[081] Devolver al flujo principal el resultado de la transacción para que el flujo principal ejecute las actualizaciones de records.
[082] Input
[083] komands_transaction (Record de u_komands_transaction)
[084] Comportamiento
[085] Espera la respuesta final en u_komands_transaction.response mediante Wait for Callback. Se configura tiempo de espera en una propiedad de Sistema.
[086] Si corresponde, ejecuta reconciliación mediante endpoint de Estado de Transacción - Komands.
[087] Retorna un resultado estándar:
[088] status (COMPLETED, ROLLBACK… )
[089] En caso de que el status no sea completed, hay que actualizar Core Use Case y Customer Order con los errores correspondientes
[090] Cuando la categoría de los errores son: RATE_LIMIT, SSH_ERROR, CLI_TIMEOUT se puede reintentar la request. En este caso cambiamos el output retry a true.
[091] Reintento (retry = true): el subflujo ejecuta un ciclo de consulta de estado llamando de forma iterativa a la API Estado de Transacción - Komands (por ejemplo GET /api/v1/transaction/{uuid}) usando el txn_id como correlación, hasta obtener un estado final (Éxito o Error) o hasta alcanzar un máximo de reintentos configurable (con espera entre intentos según configuración).  Se configura la cantidad maxma de reintentos en una propiedad de sistema.
[092] Ejemplo de los updates a Core Use Case y Customer Order records:
[093] response_body (payload final)
[094] retry (true/false)
[095] Que no hace:
[096] No actualiza records de negocio (orders/service orders/tasks/etc.). Esas actualizaciones quedan en el flujo principal.

[438] Propuesta inicial Feature Flags
[439] El mecanismo de enrutamiento hacia Komands se basa en una tabla de configuración u_komands_activation, que define por combinación de VNO, producto, tecnología y operación si una transacción debe ser procesada por Komands o por BluePlanet.
[440] Evaluación de activación por transacción
[441] Cada vez que una API recibe una solicitud, consulta u_komands_activation mediante un Script Include.
[442] Si existe un registro activo para la combinación consultada (u_is_active = true), la transacción se dirige a Komands (por ejemplo, insertando en u_assignment_v2).
[443] Si no existe un registro activo, la transacción se procesa por la ruta tradicional en BluePlanet (por ejemplo, insertando en u_assignment).
[444] Kill switch global
[445] Existe un kill switch global u.komands.enabled que permite desactivar completamente el uso de Komands en cualquier momento, forzando que todo el tráfico vuelva a BluePlanet sin necesidad de despliegues ni cambios de código.
[446] Consideraciones de diseño
[447] Este mecanismo está pensado como una solución temporal para habilitar una migración progresiva, por lo que no se prioriza un diseño altamente escalable. En un escenario permanente, sería recomendable modelar la activación de cada dimensión (VNO, producto, tecnología, operación) en estructuras separadas o más normalizadas, permitiendo evaluaciones más eficientes y flexibles. El enfoque actual favorece la simplicidad operativa y la capacidad de rollback inmediato.
